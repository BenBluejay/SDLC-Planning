# Phase 1: Business Analysis (商業分析階段)

> **RUP Phase**: Inception  
> **RUP Disciplines**: Business Modeling, Requirements  
> **里程碑**: Lifecycle Objectives (LCO) — 利害關係人對範圍達成共識  
> **關鍵決策**: Go / No-Go 決策

## 階段目的

在系統開發前，透過結構化的商業分析方法，確保我們解決的是**正確的問題**，並建立明確的**商業價值**與**系統範圍**。

**核心問題**：
- 企業真正的痛點是什麼？
- 我們要達成什麼商業目標？
- 解決這些問題的價值有多大？
- 系統應該做什麼、不做什麼？

## 輸入 (Inputs)

- 專案發起需求（來自高層或業務部門）
- 利害關係人清單
- 現有業務流程文件（如果有）
- 相關系統文件或資料

## 方法論：P/G/O/W 框架

本階段採用 **P/G/O/W 框架**（Problem → Goal → Opportunity → Way），這是一個結構化的商業分析流程，確保從痛點到解決方案的完整追溯性。

```
P (Problem)      痛點探索    → 發現並結構化描述企業問題
    ↓
G (Goal)         目標定義    → 將問題轉化為可量化的商業目標
    ↓
O (Opportunity)  機會映射    → 識別解決問題的系統功能機會
    ↓
W (Way)          方案/路線   → 定義 MVP 範圍與 User Stories
```

詳細方法請參考：[`../03-Methodology-Guides/BA-PGOW-Framework.md`](../03-Methodology-Guides/BA-PGOW-Framework.md)

## Part A: 活動規範 (Activity Specification)

### A.1 為什麼需要商業分析階段

**價值**：
- **避免做錯系統** — 確保解決的是真實問題，而非臆測的需求
- **建立價值共識** — 所有利害關係人對目標與成功標準達成一致
- **控制範圍蔓延** — 明確定義 MVP 邊界，Won't 項目與 Must 項目同等重要
- **提供追溯性** — 每個功能都能追溯到一個商業目標，每個目標都對應一個痛點

**常見失敗模式**（沒有 BA 階段時）：
- 開發到一半發現需求理解錯誤，返工成本高
- 功能完成後使用者不買單，因為沒解決真正的問題
- 範圍無限擴張，永遠沒有「完成」的一天
- 上線後無法評估成效，因為沒有量化目標

### A.2 流程步驟

#### 步驟 1: 利害關係人分析與訪談

| 子步驟 | 如何執行 | 產出 |
|--------|----------|------|
| 1.1 識別利害關係人 | 列出所有會影響或被影響的人/部門/系統 | 利害關係人清單 |
| 1.2 分析權力與利益 | 使用權力-利益矩陣分類 (高權力高利益 → 密切管理) | 利害關係人分類矩陣 |
| 1.3 規劃訪談 | 優先訪談高權力高利益者，準備訪談大綱 | 訪談計畫 |
| 1.4 執行訪談 | 半結構化訪談，記錄痛點、期望、限制條件 | 訪談記錄 |

**訪談問題範例**：
- 「你在現行流程中遇到哪些問題？多久發生一次？」
- 「這些問題造成什麼影響？（時間浪費、錯誤率、成本）」
- 「如果有系統能解決這個問題,對你的工作有什麼幫助？」
- 「你認為系統必須具備哪些功能？」
- 「有哪些功能是你希望有但不是必須的？」

#### 步驟 2: P (Problem) — 痛點探索

| 子步驟 | 如何執行 | 產出 |
|--------|----------|------|
| 2.1 從訪談中萃取問題 | 將訪談記錄中的抱怨轉化為結構化問題陳述 | 初步問題列表 |
| 2.2 問題分類 | 按領域/流程/角色分類 | 分類後問題列表 |
| 2.3 問題優先序 | 以「影響度 × 頻率 × 急迫性」評分 | 優先序矩陣 |
| 2.4 撰寫問題陳述 | 每個問題一個唯一 ID (P1, P2...) 與完整描述 | 問題陳述文件 (BA-02) |

**問題陳述格式**：
```
| ID | 問題陳述 | 影響對象 | 發生頻率 | 急迫性 | 當前影響 |
|----|----------|----------|----------|--------|----------|
| P1 | 使用者透過 Email + Excel 提交申請，無統一平台 | HR、部門主管 | 每週 | 高 | 處理時間長、資料不一致 |
| P2 | 部門預算可見度為零，常超支才發現 | 財務、部門主管 | 每月 | 高 | 預算超支、需緊急調整 |
```

#### 步驟 3: G (Goal) — 目標定義

| 子步驟 | 如何執行 | 產出 |
|--------|----------|------|
| 3.1 問題轉目標 | 每個 Pn 至少對應一個 Gn | 目標-問題對照表 |
| 3.2 量化目標 | 為每個目標定義可衡量的 KPI | 量化目標表 |
| 3.3 利害關係人簽核 | 確認目標與 KPI，取得書面同意 | 簽核後的商業目標文件 (BA-03) |

**目標 KPI 範例**：
```
| 問題 ID | 商業目標 | KPI | 目標值 | 衡量方式 |
|---------|----------|-----|--------|----------|
| P1 | 建立統一申請平台 | 審批週期時間 | 縮短 50% | 系統時間戳記 |
| P2 | 即時預算可見度 | 預算超支次數 | 降至 0 次/季 | 財務報表 |
```

#### 步驟 4: AS-IS / TO-BE 流程分析

| 子步驟 | 如何執行 | 產出 |
|--------|----------|------|
| 4.1 繪製 AS-IS 流程 | 使用 Swimlane 圖標示現行流程 | AS-IS 流程圖 (BA-04) |
| 4.2 標註痛點 | 在流程圖上標記每個 Pn 發生位置 | 痛點標註的流程圖 |
| 4.3 估算時間 | 記錄每個步驟的平均處理時間與等待時間 | 時間估算表 |
| 4.4 設計 TO-BE 流程 | 描繪系統輔助後的新流程（概念層級） | TO-BE 流程圖 (BA-05) |
| 4.5 定義 RACI | Responsible / Accountable / Consulted / Informed | RACI 矩陣 |

#### 步驟 5: O (Opportunity) — 機會映射

| 子步驟 | 如何執行 | 產出 |
|--------|----------|------|
| 5.1 痛點轉機會 | 每個 Pn → 一個或多個 On（系統功能機會） | 機會清單 |
| 5.2 機會群組化 | 將相關機會群組為功能模組 | 功能模組分組 |
| 5.3 評估成本效益 | 每個機會評估：成本 / 效益 / 風險 | 評估表 |
| 5.4 產出機會地圖 | 完整的 Problem → Opportunity 映射 | 機會地圖 (BA-06) |

**機會地圖範例**：
```
P1 (格式混亂) → O1: 結構化線上表單
              → O2: 即時欄位驗證
P2 (預算盲點) → O3: 預算儀表板
              → O4: 提交時自動檢查餘額

功能群組:
  [申請模組]   O1 + O2
  [預算模組]   O3 + O4
```

#### 步驟 6: User Story Mapping 工作坊

**目的**：將 Opportunities 組織成視覺化的使用者旅程，劃分 MVP 與後續 Release 邊界。

**工作坊形式**（2-4 小時）：
- **參與者**：產品負責人、領域專家、利害關係人、UX 設計師、少數開發代表
- **工具**：便利貼 + 白板（或 Miro / Mural）
- **產出**：User Story Map (BA-08) + User Story Survey (BA-09) + Release Plan (BA-10)

| 子步驟 | 如何執行 | 產出 |
|--------|----------|------|
| 6.1 識別 Activities | 從 TO-BE 流程圖萃取使用者的高層活動（黃色便利貼） | Activity 清單 |
| 6.2 分解 User Tasks | 每個 Activity 下的具體任務（藍色便利貼） | User Task 清單 |
| 6.3 撰寫 User Stories | 每個 Task 的實作故事（粉色便利貼，輕量版） | User Story 草稿 |
| 6.4 橫向排列（User Journey） | 按使用者流程從左到右排列 Activities | Story Map 橫軸 |
| 6.5 縱向排列（Priority） | 按優先序從上到下排列 Stories | Story Map 縱軸 |
| 6.6 劃分 Release 切線 | 畫橫線定義 MVP / Release 2 / Release 3 邊界 | Release Plan (BA-10) |

### A.3 設計原則與最佳實踐

#### 原則 1: User Story Mapping 視覺化優先，Survey 為主要需求規格格式

**為什麼使用 User Story Mapping**：
- **視覺化使用者旅程** — 利害關係人更容易理解系統全貌
- **優先序自然浮現** — 縱軸排列讓 MVP 與後續 Release 邊界清晰
- **避免遺漏步驟** — 橫軸的使用者流程幫助完整性檢查
- **便於討論與共識** — 便利貼可移動、可調整，工作坊互動性強

**為什麼 BA 階段不寫 Acceptance Criteria**：
- **BA 階段關注範圍與價值** — 確認「要做什麼」與「為什麼」
- **SA 階段關注實作細節** — 透過 DDD 分析後才能寫出精確的 AC
- **避免過早設計** — 在理解領域模型前寫 AC 容易出錯，需要返工
- **保持輕量** — User Story Survey 讓利害關係人快速檢視與調整優先序

**User Story Survey 用途**：
- 範圍確認（與利害關係人達成共識）
- 優先序排列（MoSCoW 分類）
- Release 規劃（MVP / R2 / R3）
- 追溯性（Story → Opportunity → Goal → Problem）

**完整 User Story 規格（含 AC）在 Phase 2 補充**：
- Phase 2 透過 Event Storming 識別 Domain Events
- 透過 Bounded Context 劃分確認 Story 所屬的語義邊界
- 透過 Aggregate 設計理解業務規則與約束條件
- 此時補充 Given/When/Then AC 才精確且穩定

**何時補充其他格式**：
- 複雜業務規則 → 加入決策表 (Decision Table)
- 狀態機流程 → 加入狀態轉移圖
- 複雜 UI 互動 → 加入 Mockup / Wireframe

#### 原則 2: 每個功能都能追溯到痛點

維持追溯鏈：
```
Problem (P1) → Goal (G1) → Opportunity (O1) → User Story (US-01)
```

如果一個 User Story 無法追溯到 Opportunity，該 Opportunity 無法追溯到 Goal，該 Goal 無法追溯到 Problem，則可能超出範圍。

#### 原則 3: Won't 項目與 Must 項目同等重要

明確記錄「不做什麼」避免範圍蔓延：
```
| Won't 項目 | 理由 | 替代方案 |
|------------|------|----------|
| 合規稽核引擎 | 複雜度高，Phase 1 不做 | 手動稽核，Phase 2 規劃 |
| 多幣別支援 | 目前僅國內使用 | 單一幣別，未來擴充 |
```

#### 原則 4: Ubiquitous Language 從 BA 開始建立

從訪談開始收集領域術語，建立詞彙表：
```
| 術語 | 定義 | 拒絕的別名 |
|------|------|-----------|
| HeadcountRequest | 部門主管提出的人力需求申請 | 「HC 申請」、「開缺」 |
| ApprovalStatus | 審批流程狀態 (DRAFT/SUBMITTED/...) | 「階段」、「狀態」 |
```

這個詞彙表會延續到 Phase 2 的 DDD 設計。

## 常見陷阱 (Pitfalls)

### ❌ 跳過 BA 直接寫程式

**症狀**: 「需求很簡單，直接開始做就好」  
**後果**: 開發到一半發現理解錯誤，或做出來的系統使用者不買單  
**解法**: 即使是小專案也至少要完成 P/G/O/W 的簡化版本

### ❌ 寫了 Use Case 就不寫 User Story

**症狀**: 認為 Use Case 已經是需求規格了  
**後果**: Use Case 偏技術流程描述，缺少使用者價值與驗收標準  
**解法**: **User Stories 為主要需求規格格式**，Use Case 僅作為補充說明複雜流程

### ❌ User Story 沒有 Acceptance Criteria

**症狀**: Story 只有 As-a / I-want / So-that，沒有 AC  
**後果**: 開發者不知道「做到什麼程度算完成」，測試者不知道如何驗收  
**解法**: 每個 User Story 必須有可測試的 AC，而且要在 BA 階段就寫好

### ❌ 範圍無限擴張

**症狀**: 「這個功能也不錯」、「那個也可以加」  
**後果**: 永遠沒有「完成」的一天，團隊疲憊不堪  
**解法**: 嚴格執行 MoSCoW 分類，Won't 項目明確記錄並與利害關係人溝通

### ❌ 目標沒有量化

**症狀**: 「改善使用者體驗」、「提升效率」  
**後果**: 上線後無法評估是否成功  
**解法**: 所有目標必須有可衡量的 KPI，例如「審批時間縮短 50%」

### ❌ Ubiquitous Language 不一致

**症狀**: 訪談時說「開缺」，User Story 寫「招募需求」，程式碼用「RecruitmentRequest」  
**後果**: 溝通成本高，容易誤解  
**解法**: 建立詞彙表，強制使用統一術語

## 階段檢查清單 (Checklist)

在進入 Phase 2 之前，確認以下項目：

- [ ] 已訪談所有高權力高利益利害關係人
- [ ] 問題陳述清晰且已優先序排列 (BA-02)
- [ ] 每個商業目標都有量化的 KPI (BA-03)
- [ ] AS-IS 流程圖已繪製並標註痛點 (BA-04)
- [ ] TO-BE 流程圖已繪製 (BA-05)
- [ ] 機會地圖完整，每個問題都有對應機會 (BA-06)
- [ ] MoSCoW 分類完成 (BA-07)
- [ ] User Story Map 工作坊完成，視覺化 Story Map 已產出 (BA-08)
- [ ] User Story Survey 已從 Story Map 匯出成表格 (BA-09)
- [ ] Release Plan 已定義 MVP / R2 / R3 邊界 (BA-10)
- [ ] 所有 MVP User Stories 都已納入 Survey，Actor / Purpose 欄位完整
- [ ] 非功能性需求已定義 (BA-11)
- [ ] 驗收測試計畫已撰寫 (BA-12)
- [ ] Ubiquitous Language 詞彙表已建立（初版）
- [ ] 利害關係人已簽核範圍與目標
- [ ] **Go / No-Go 決策已完成** ← Lifecycle Objectives 里程碑
